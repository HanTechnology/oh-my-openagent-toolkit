## Incident Examples

### Example 1: Database Performance Incident (Critical)

**INCIDENT-201: Database Query Timeout Causing Login Failures**

**Severity**: CRITICAL

**Timeline**:
- **Detected**: 2025-01-20 14:35 UTC (Monitoring alert: Database response time >5s)
- **Acknowledged**: 2025-01-20 14:37 UTC (2 minutes)
- **Mitigated**: 2025-01-20 14:52 UTC (17 minutes) - Cache layer activated
- **Resolved**: 2025-01-20 16:15 UTC (1.67 hours) - Database query optimized

**Impact**:
- **Users Affected**: 1,250 users (15% of active users)
- **Services Down**: Login service, User profile API
- **Duration**: 1 hour 40 minutes
- **Data Loss**: None
- **Financial Impact**: ~$500 (estimated revenue loss)

**Root Cause**:
Unoptimized database query introduced in v1.4.2 deployment. The query performed a full table scan on 5M+ user records without proper indexing, causing response times to spike from 50ms to 8+ seconds.

**Contributing Factors**:
1. Missing database query performance test in CI/CD pipeline
2. No query plan analysis before deployment
3. Staging database had only 10K records (didn't reproduce issue)

**Detection**:
- **How Detected**: Automated monitoring alert (Database response time >5s)
- **Detection Lag**: 2 minutes from incident start
- **Alert Fired**: Yes - "Database performance degradation"

**Response Actions**:
1. **14:37 UTC**: Incident acknowledged, backend-nestjs skill investigating
2. **14:40 UTC**: Identified slow query in application logs
3. **14:45 UTC**: Activated Redis cache layer to bypass database for common queries
4. **14:52 UTC**: Service partially restored (80% of requests now served from cache)
5. **15:10 UTC**: Created missing database index on user_id + status columns
6. **15:45 UTC**: Deployed optimized query with proper indexing
7. **16:00 UTC**: Monitoring confirms response times back to normal (<100ms)
8. **16:15 UTC**: Incident declared resolved

**Mitigation Strategy**:
Immediate: Activated Redis cache layer to serve frequently accessed user data, bypassing the slow database query for 80% of login requests.

**Permanent Fix**:
1. Added composite index on users table (user_id, status, last_login)
2. Rewrote query to use indexed columns and avoid full table scan
3. Query response time improved from 8000ms to 45ms (99.4% improvement)

**Lessons Learned**:
1. Staging environment must mirror production data volume (10K vs 5M records gap)
2. All database queries need performance testing with production-scale data
3. Query plan analysis should be part of code review process
4. Need proactive database performance monitoring, not just reactive alerts

**Action Items**:
- [x] Add database query performance tests to CI/CD pipeline - backend-nestjs - Completed: 2025-01-21
- [x] Implement query plan analysis in pre-deployment checks - devops-deployment - Completed: 2025-01-22
- [x] Populate staging database with production-scale test data - systemdev-specialist - Completed: 2025-01-23
- [ ] Add proactive slow query monitoring (>500ms warning threshold) - devops-deployment - Due: 2025-01-27

**Related Issues**: None (first occurrence)

**Post-Mortem**: workspace/docs/post-mortems/2025-01-20-database-performance.md

**Status**: CLOSED ✅

### Example 2: Security Incident (High)

**INCIDENT-202: Unauthorized API Access Attempts**

**Severity**: HIGH

**Timeline**:
- **Detected**: 2025-01-18 09:22 UTC
- **Mitigated**: 2025-01-18 09:45 UTC
- **Resolved**: 2025-01-18 11:30 UTC

**Impact**:
- **Users Affected**: 0 (no successful breaches)
- **Services Affected**: API rate limiting triggered
- **Attack Volume**: 50,000+ requests in 15 minutes
- **Data Compromised**: None

**Root Cause**:
Brute-force attack attempting to exploit API endpoint with weak rate limiting (1000 req/min per IP). Attacker used distributed IPs to bypass rate limits.

**Response Actions**:
1. **09:22 UTC**: Security alert fired (unusual API traffic pattern)
2. **09:25 UTC**: backend-nestjs + devops-deployment investigating
3. **09:30 UTC**: Confirmed malicious traffic, identified attack pattern
4. **09:35 UTC**: Temporarily blocked attacking IP range
5. **09:45 UTC**: Implemented stricter rate limiting (100 req/min per endpoint)
6. **10:15 UTC**: Added CAPTCHA for repeated failed login attempts
7. **11:00 UTC**: Deployed Web Application Firewall (WAF) rules
8. **11:30 UTC**: Attack mitigated, normal traffic restored

**Permanent Fix**:
1. Enhanced rate limiting: Global + Per-endpoint + Per-user limits
2. Added CAPTCHA after 3 failed login attempts
3. Implemented WAF with DDoS protection
4. Added IP reputation scoring

**Lessons Learned**:
1. Rate limiting needs multiple layers (global, endpoint, user)
2. Security monitoring should detect unusual patterns, not just failures
3. CAPTCHA should be proactive, not reactive

- [x] Add CAPTCHA to all authentication endpoints - frontend-nextjs - Completed: 2025-01-19
- [x] Deploy WAF with DDoS protection - devops-deployment - Completed: 2025-01-20
- [ ] Conduct security audit of all public endpoints - qa-testing - Due: 2025-01-30
